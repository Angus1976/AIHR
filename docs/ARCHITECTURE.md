# 架构与需求分析（职得 Jobde）

## 1. 分析目标

在 PRD v1.1 基础上，形成**自洽**的工程视图：谁用什么终端、依赖哪些外部系统、数据如何分区、Docker 与微信小程序各自负责什么，以及仍依赖产品/法务拍板的**开放问题**。

## 2. 干系人与终端矩阵

| 干系人 | 主终端 | 次要终端 | 备注 |
|--------|--------|----------|------|
| 求职者 | 微信小程序 | — | 主路径：浏览、服务、签约、支付 |
| 就业老师 | Web 管理端（全功能优先） | 小程序精简配置 | 移动端以「只读/开关」为主，避免复杂编排 |
| 企业对接人 | Web（未来可扩展企业端） | MVP 多为**代操作**，企业或仅用 H5/外链确认 | PRD：企业端 MVP 以平台代发布为主 |
| 伙伴机构 | Web 入驻表单 + 审核流 | — | 资质与课程上架 |
| 管理员 | Web | — | AI 面板、品牌白标、权限、合规模板 |

**自洽结论**：至少两条前端交付物——**小程序**与 **Web Admin**；共用同一套 **Backend API** 与 **领域模型**。

### 3.1 UI 设计系统（2026-04-25）

- **视觉定位**：现代、大方、可信的人才服务平台；强调“就业服务工作流”和“运营留痕”，避免泛 AI 产品常见的蓝紫渐变、霓虹发光、emoji 图标和无意义数据堆砌。
- **色彩**：暖纸色背景、深林绿主色、铜橘强调色、鼠尾草绿成功态；所有状态色、边框和按钮都从 token 派生，不在页面内硬编码新色相。
- **字体**：Web 管理端采用 `Sora + Noto Sans SC / PingFang SC`；小程序使用微信端稳定中文字体栈，保持中文可读性优先。
- **形态**：卡片为纸面质感、细边框、克制阴影；按钮为深林绿实心主按钮，弱按钮使用纸白底 + 细边框；状态标签统一圆角胶囊。
- **动效**：仅使用轻量微交互（hover 上浮、focus ring、160-240ms 过渡），不做夸张动画和渐变背景。
- **一致性约束**：新增页面必须复用 `theme.css` / `app.wxss` token；禁止回退到 `#003087`、蓝紫渐变、左侧彩色边框卡片等旧样式。

### 3.2 「AI 服务引擎」在工程中的落点

- **配置面**：提示词/Skills/模型路由/环节绑定 → 存 DB + 版本表 +（可选）特性开关服务。
- **运行面**：异步任务队列（长耗时评估、生成报告）+ 同步接口（短问答）；全链路 **request_id**、**tenant_id**、**operator_id**（代操作时必填）。
- **观测面**：调用日志、脱敏输出、用户反馈标签；供面板展示与告警。

### 3.3 代操作（就业老师）模型

每条关键业务动作建议携带：

- `performed_by`：实际操作账号；
- `on_behalf_of`：被代理的用户/企业实体；
- `reason`（可选）：工单号或备注。

满足审计与 PRD「留痕」要求，且与 PIPL 问责一致。

## 4. 多租户与白标

PRD 要求一键换品牌。建议**首版即带租户表**：

- `Tenant`：名称、Logo URL、主题色、欢迎语、功能开关；
- 请求从域名或 Header（管理端）/ 小程序 `appId` 映射到租户；
- 静态资源（Logo）走对象存储 + CDN，配置仅存 URL 与色值。

## 5. 部署模型：Docker vs 微信小程序

### 5.1 Docker 负责什么

| 组件 | 是否适合容器化 | 说明 |
|------|----------------|------|
| PostgreSQL | 是 | `docker-compose` 本地与单机生产均可；生产建议托管 RDS |
| Redis | 是 | 会话、队列、限流 |
| API 服务 | 是 | 构建镜像，环境变量注入密钥 |
| Admin Web | 是 | 静态资源用 Nginx 镜像，或 SSR 容器 |
| 小程序 | **否** | 源码在微信开发者工具编译，**上传**至微信平台；CI 可用 headless/官方 cli 打 npm 包，但运行时不在自有 Docker |

**自洽结论**：「支持 Docker 部署」= **后端与 Web 管理端及依赖中间件** 容器化；小程序走 **微信侧发布流水线**，仅 **API 基地址** 指向你的 Docker/K8s 暴露域名。

### 5.2 推荐环境分层

| 环境 | 用途 |
|------|------|
| local | `docker compose` + 本机微信开发者工具，`request` 合法域名为内网穿透或局域网调试 |
| staging | HTTPS + 正式类小程序体验版/开发版，联调支付与签章沙箱 |
| production | HTTPS + 小程序正式版；密钥走密钥管理，禁止进镜像 |

## 6. 安全与合规（工程 checklist）

- **传输**：全站 HTTPS；HSTS（Web）。
- **鉴权**：小程序 `code2session`；管理端建议 OIDC 或账号体系 + RBAC。
- **个人信息**：字段级权限、展示脱敏、导出审批、留存策略可配置。
- **同意**：单独同意记录（版本号 + 文案 hash + 时间）。
- **签章**：合同 PDF hash、签署状态、与订单/节点绑定；密钥放服务端。
- **日志**：不落敏感明文；PII 访问写审计日志。

## 7. MVP 范围与 Phase 2（与 PRD 对齐）

**Phase 1（MVP）**：6 环节 AI 核心、求职者小程序、企业岗位**代发布**、伙伴入驻基础、AI 配置面板、收费与协议流（含电子签接口位）。

**Phase 2**：ATS、企业自助发布、更多伙伴类型、高级分析。

## 8. 工程决策记录（2026-04-24 与产品方确认）

| 主题 | 决策 | 工程落点 |
|------|------|-----------|
| 后端技术栈 | **无强语言偏好**；当前仓库 MVP 采用 **Node.js + NestJS + Prisma** | `apps/api` |
| 腾讯系绑定 | **深度绑定**：小程序登录走 `jscode2session`；支付/电子签/会议等按迭代接入腾讯系或官方 API | `WechatService`、集成占位服务 |
| 企业端入口 | 企业客户以 **HTTP Web** 为主（MVP 含平台代操作 + 后续企业门户） | `apps/admin-web` + 未来可拆 `enterprise-web` |
| 微信支付 | **平台单一收款**；已具备商户号；类目 **人才服务** | `WechatPayService` 占位 → 后续 JSAPI/小程序支付 |
| 电子签 | **未选定**；原则 **云厂商（腾讯）**；先预留接口 | `TencentSignatureService` 占位 |
| LLM | **不限混元**；支持 **全球厂商** API + 混元；路由可配置 | `LlmRouterService` 占位 + 后续 DB 路由 |
| 面试/入职确认 | **允许平台方代点**；认证支持 **邮箱或短信**（管理端登录已种子邮箱密码；短信后续接腾讯云 SMS） | `EnterpriseConfirmation` + `AuditLog` |
| 小程序 | **原生** | `apps/miniprogram` |

### 8.1 产品迭代优先级（工程执行顺序）

1. **平台代发布岗位与运营闭环**：企业列表、岗位草稿/发布、审计留痕、管理端操作台。  
2. **微信支付 v3（平台单一商户）**（已接基线）：`Order` 表、`POST /payments/wechat/jsapi`（配置齐全则调微信统一下单并返回 `wx.requestPayment` 参数；否则 stub）、`POST /payments/wechat/notify`（解密成功则置 `PAID`）；**生产须补**：回调签名校验、平台证书、幂等与对账任务。  
3. **腾讯云短信**（已接基线）：`SmsOtp` 落库 + HMAC 哈希、`TencentSmsService`（配置齐全则 `SendSms`）、`POST /auth/sms/send|verify`、企业联系人代发 `POST /enterprises/:id/contact-sms/send`；**生产须补**：图形/行为验证码防刷、IP 限流、模板与签名合规审核。

### 8.2 仍待法务/运营细化的开放项

- 小程序**具体类目与资质材料**与接口权限的最终清单。  
- 职后服务「离职仍享」对**订阅计费与退款**的条款。  
- PIPL 相关**同意文案版本**与 PIA 流程责任部门。
